# 01: 流式 markdown 渲染 O(N²) 致对话输出卡死

- 状态:`ready-for-agent`
- 关联:`mdToHtml`(server markdown)、`TextBlock`(web ChatPanel)、`useChat` 的 `text_delta` 分支
- 关联 ADR:`docs/adr/0002-layer1-contract.md`(`mdToHtml` 零依赖、前后端共用);CLAUDE.md 零依赖风格

## 背景 / 根因

desktop 对话流式输出中途,界面无法点击、无法打开 DevTools,但窗口仍可拖动缩放;macOS 报一条 `error messaging the mach port for IMKCFRunLoopWakeUpReliable` 后无任何日志;重启对话必现。代码路径分析坐实根因(运行时复现:启动后发一条要求长输出的对话,如「写一篇 3000 字的文章」):

- agent 每个 token 触发一条 `text_delta` WS 消息。
- `useChat` 的 `onmessage` `text_delta` 分支每次 `setMessages(prev => prev.map(...))`,把当前 assistant 末尾 text 段拼成 `last.text + delta`,`seg.text` 引用每 delta 必变。
- `TextBlock` 用 `useMemo(() => mdToHtml(text), [markdown, text])`,缓存失效 -> **每个 delta 都对累积全文同步跑一次 `mdToHtml`**。
- `mdToHtml` 是 O(全文长度) 的同步纯字符串解析;N 个 delta -> 总计算量 **O(N²)**。
- 长输出(几千 token)打满渲染进程主线程 -> 内容区卡死;主进程未阻塞故窗口可拖动。
- `IMKCFRunLoopWakeUpReliable` 是渲染进程卡死后系统输入法探测无响应的**副产物,非根因**,渲染卡死解决后自然消失。
- 代码在 web 层共享,浏览器形态(`make dev`)长输出同样会卡。

## What to build

方案 4:块级缓存 + 末块节流。流式时仍**实时显示 markdown 格式**(不降级纯文本),保持与 `mdToHtml` 同源,不引入 `react-markdown` / `llm-ui` 等新依赖。

端到端行为:

- 从 `mdToHtml` 的块扫描逻辑抽出一个 `splitBlocks(text): Block[]` 函数,识别块(段落 / 代码块 / 列表 / 表格 / 引用 / 标题 / hr)及块边界,每个块标记 `complete`(已闭合)或 `partial`(末尾正在写)。**块边界判断须与 `mdToHtml` 的 while 循环一致**(代码块 fence、列表、表格、引用、标题、hr、段落空行),否则缓存 key 错位。`mdToHtml` 本体的渲染输出**不变**,只抽出块识别复用。
- 流式渲染:**每个 text 段(`Segment { kind: 'text' }`)独立做块级渲染,不合并消息内多个 text 段**(保留 text/tool 段时序)。对该段的 `seg.text` 跑 `splitBlocks`:`complete` 块以块原文为 key 缓存 `mdToHtml(块)`,命中复用;只有最后一个 `partial` 块每次更新重算 `mdToHtml(该块)`。流式时只有末尾 text 段在增长(`text_delta` 追加到末尾 text 段),非末尾 text 段全部 `complete`、整体缓存。O(N²) -> ~O(N)。
- **块级 DOM**:每个块渲染为独立子元素(`React.memo` + 稳定 key),`complete` 块 html 引用稳定 -> React 不重渲染、DOM 节点不动;只 `partial` 块更新。**禁止做成单 `div` + 全文 `dangerouslySetInnerHTML`**(否则 DOM 重建仍 O(N²),块缓存白做)。
- 末块节流:对 `partial` 块的 `mdToHtml` 重算用时间窗口 / raf 节流(约 80–120ms 一次),防超长单段落(无空行万字级)退化;`done` 事件强制算一次最终结果。
- smooth scroll 节流:`ChatPanel` 现有 `useEffect([messages])` 的 `scrollTo({behavior:'smooth'})` 每 delta 触发,长输出堆积。改为 raf / 时间窗口节流,或流式期间 `instant`、`done` 后 `smooth`。

`useChat` 的 `text_delta` 累积逻辑(拼接 delta 到末尾 text 段)**不变**,块级处理放在渲染层。`MessageBubble` 用 `React.memo` 包裹,使每 delta `setMessages` 只有流式中的末尾消息重渲染、历史消息跳过 diff;其内部其他渲染(user 纯文本、`ToolChip`、system 消息)不变。

## Acceptance criteria

- [ ] 长输出流式过程(让 agent 写 3000+ 字)界面不卡、可点击交互、能打开 DevTools。
- [ ] 流式过程中实时显示 markdown 格式(非纯文本降级)。
- [ ] `done` 后最终 markdown 与现有 `mdToHtml` 输出一致(块级缓存不改变渲染结果)。
- [ ] 纯浏览器形态(`make dev`)长输出同样不卡。
- [ ] 超长单段落(无空行万字级)不卡死(末块节流兜底)。
- [ ] smooth scroll 流式期间不堆积卡顿。
- [ ] `MessageBubble` 用 `React.memo` 包裹;长对话(多条历史消息)流式输出时,历史消息不进入重渲染(React DevTools Profiler 验证)。
- [ ] 不引入 `react-markdown` / `llm-ui` / `assistant-ui` / `shiki` 等新依赖(`package.json` 依赖不变)。
- [ ] chat 渲染仍用 `mdToHtml`,与 wiki 文章渲染同源。
- [ ] 新增 `splitBlocks` 单元测试:块边界识别与 `mdToHtml` 一致(段落 / 代码块 / 列表 / 表格 / 引用 / 标题 / hr 各至少一例,含未闭合的 `partial` 末块)。
- [ ] `make typecheck` + `make lint` + `make format` 通过。
- [ ] `npm test` 通过,现有测试无回归。

## Out of scope

- 不引入 `react-markdown` / `llm-ui` / `assistant-ui` / `shiki`(保持零依赖 + 渲染同源)。
- 不改 `mdToHtml` 的渲染输出行为(只抽 `splitBlocks` 复用块识别,解析结果不变)。
- 不做虚拟化(视口外块不挂载)-- 十万字+ 才需要,当前 chat 场景过重。
- 不改 server WS 协议 / `text_delta` 事件结构。
- 不改 `useChat` 的 WS 重连 / 退避 / 清理逻辑(已健全)。
- 不处理 `IMKCFRunLoopWakeUpReliable` 错误本身(副产物,渲染卡死解决后自然消失)。
